# Telegram Immediate Reply Policy

Owner: Codex / Lead Architect

Last updated: 2026-05-17T04:12Z

## Goal

Telegram should never feel silent.

Every owner command must get:

1. An immediate bot acknowledgement from `poll-loop.mjs`.
2. For direct commands, an immediate bot answer.
3. For Codex-required commands, an immediate `codex exec` run from the Telegram poll loop.
4. A final Telegram answer from the `codex exec` output.
5. A durable record in `COMMAND_QUEUE.md` and `COMMAND_RESULTS.md`.

If Codex is already doing unrelated work, Telegram commands should not steer the active work unless they conflict with it. Non-conflicting commands are handled at the next safe checkpoint or heartbeat.

## What Can Be Immediate

The bot can answer these without waiting for Codex:

- `/help`
- `/status`
- `/progress`
- `/workers`
- `/worklist`
- `/blockers`
- `/links`
- Simple arithmetic through `/codex`, such as `/codex 3*5`

The bot also immediately acknowledges queued commands:

- `/codex <task>`
- `/decide <decision>`
- `/priority <item>`

## What Requires Codex

Commands that require reasoning, product decisions, code edits, QA updates, or project-state changes are sent to Codex immediately with `codex exec`.

Codex processes Telegram commands without owner steering when they do not conflict with active work. If a command changes priority, asks to stop/revert current work, or touches secrets/accounts/legal/payment setup, Codex treats it as a steering or blocker event.

Codex writes the final owner-facing result to:

```text
docs/telegram/COMMAND_RESULTS.md
```

The running `poll-loop.mjs` sends the `codex exec` result back to Telegram immediately. It also drains older pending results from `COMMAND_RESULTS.md` every polling cycle.

The default Telegram polling interval is 10 seconds. This is lightweight: it is a Telegram Bot API network poll and does not spend Codex tokens by itself. Codex tokens are used only when the queued command is processed by Codex.

## Current Limitation

If Telegram only receives `Queued for Codex` and never receives the final answer, the running poll loop is stale and should be restarted. Current scripts should say `Codex is processing this now.`

Fix:

1. Stop the old Telegram poll loop.
2. Start `scripts/telegram/poll-loop.mjs` with `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID`.
3. The loop will send pending `COMMAND_RESULTS.md` answers automatically.

## No Secret Rule

Do not hardcode the bot token or chat ID into source files. Runtime environment variables are required.

## Operational Decision

For now, final replies are not generated inside the Telegram bot itself. They are generated by Codex during heartbeat or active work, then sent by the poll loop. This keeps product/code decisions under Codex control and avoids a second external agent with unclear authority.
